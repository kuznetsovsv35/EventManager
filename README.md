# Event Manager Web API application
Продвинутая разработка на C# и .NET. Проектная работа.

# Работа над проектом.
## Спринт 1.
1. Создан каркас приложения (базовая структура).
2. Создание модели событияю.
3. Реализация хранилища события в памяти на основе TF Core.
4. Реализация интерфейса IEventService и объектов DTO.
5. Реализация EventsControler (API контролера).
6. Реализация атрибута валидации класса модели входящих данных.
7. Подготовка к публикации проекта (мелкие правки и форматирование,, документирование кода).

## Спринт 2.
1. Создан middleware для глобального обработчика ошибок.
2. При ошибках сервис возвращает единный формат ответ JSON-объект со схемой ProblemDetails.
3. Реализовано логирование исключительных ситуаций (осуществляется глобальным обработчиком).
4. Метод GET /Events дополнены параметрами фильтрации и разбивки массива данных на страницы.
5. Фильтрация данных построена на основе создания выражений в зависимости от параметров фильтрации.
6. Фильтрация реализована в отдельном самостоятельном сервисе.
7. Добавлен самостоятельный сервис разбивки набора данных, принимающий в качестве параметра размер страницы и номер запрашиваемой страницы.
8. Дополнена документация Swagger.
9. Написаны тесты отдельно для сервисов фильтрации и пагинаци, тесты манипулирования событиями, интегральные тесты.
10. Подготовка к публикации проекта (мелкие правки и форматирование, документирование кода).

# Сборка и запуск проекта.
1. Клонирование репозитория:
   ```bash
   git clone -b sprint-2 git@github.com:kuznetsovsv35/EventManager.git
   ```
2. Перейти в паку "EventManager":
   ```bash
   cd EventManager
   ```
3. Сборка проекта (целевой фреймворк net10.0):
   ```bash
   dotnet build
   ```
4. Запуск тестов.
   В проекте предусмотрены юнит-тесты сервисов фильрации и пагинации, тесты манипуляции с событиями, а также интегальные тесты функциональности сервиса управления событиями. Все тесты распределены по категорям.
   Таблица 1. Категории фильтров.
   | Категория | Описание |
   |-----------|----------|
   | Filters | В категорию входят юнит-тесты и нтегральные тесты сервиса фильтрации. |
   | Paginator | В категорию входят юнит-тесты и интеральные тесты сервиса пагинации. |
   | Service | В категормю входят тесты функций CRUD |

   Для запука тестов выполнить одну из команд:
   ```bash
   # Запуск всех тестов.
   dotnet test

   # Запуск тестов сервиса фильтрации.
   dotnet test --filter "Category=Filters"

   # Запуск тестов сервиса разбивки на страницы.
   dotnet test --filter "Category=Paginator"

   # Запуск тестов сервиса CRUD функциональности.
   dotnet test --filter "Category=Service"
   ```

6. запуск приложения:
   ```bash
   # Вариант 1.
   dotnet run --project EventManager
   
   # Вариант 2 (убедиться, что у файла скрипта есть атррибут выполнения).
   ./run.sh
   ```
7. В браузере открыть: http://localhost:5108/swagger
8. На странице можно увидеть описание API и попробовать его в деле.

# Описание API.

Таблица 2. Методы API, данные запросов/ответов, возможные ответы.
| Метод | URI | Действие | Данные запроса | Ответ | Данные ответа | Возможные ошибки |
|-------|-----|----------|----------------|-------|---------------|------------------|
|GET|/events|Получить события.|см. таблицу 3 ниже.|OK|см. таблицу 6|BAD_REQUEST|
|GET|/events/{id:uuid}|Получить событие по идентификатору id.|id - ID события|OK|см. таблицу 5|NOT_FOUND|
|POST|/events|Создать событие.|см. таблицу 4|CREATED|см. таблицу 5|BAD_REQUEST|
|PUT|/events/{id:uuid}|Обновить данные о событии.|id обытия, см. таблицу 4|OK|см. таблицу 5|BAD_REQUEST, NOT_FOUND|
|DELETE|/events/{id:guid}|Удалить событие по идентификатору id.|id - ID удаляемого события|OK|см. таблицу 5|NOT_FOUND|
### Примечание: 
   В случае ошибки тело ответа представляет стандартну структуру ProblemDetails.


Таблица 3. Описание параметров запроса событий.
| Идентиикатор | Тип | значение по-умолчанию | Описание |
|---|---|---|---|
|title|string|null|Подстрока, включенная в заголовок события.|
|startAt|datetime|null|Дата не раньше котрой начинаются события. Отсутствия значение - момент начала не ограничен.|
|endAt|datetime|null|Да непозже которой заканчиваются события. Отсутствие значения - момент окончания не ограничен.|
|page|int|1|Номер запрашиваемой страницы данных. Отсутствие значения - первая страница.|
|pageSize|int|10|Размер страницы в элементах событий. Отсутствие значение - 10 объектов на страницу.|

Таблица 4. Структура EventInputData - входящие данные о событии для создания или обноления события.
| Поле | Тип данных | Описание |
|---|---|---|
|title|string|Заголовок события.|
|description|string|Необязательное описание события.|
|startAt|string($date-time)|Момент начала события.|
|endAt|string($date-time)|Момент окончания события.|

Таблица 5. Структура EventOutputData - информация о событии, возвращаемая сервисом.
| Поле | Тип данных | Описание |
|---|---|---|
|title|string|Заголовок события.|
|description|string|Описание события (если заполнено).|
|startAt|string($date-time)|Момент начала событи.|
|endAt|string($date-time)|Момент окончания события.|
|id|string($uuid)|Уникальный идентификатор записи о событии.|

Таблица 6. Структура PaginateResult - результат запроса массива событий, сформированного в результате фильтрации и пагинации.
| Поле | Тип данных | Описание |
|---|---|---|
|totalCount|int|Общее количество элементов в наборе|
|values|EventOutputData[]|Массив объектов описаний событий, размещенных на запрошенной странице.|
|pageNumber|int|Номер запрошенной страницы.|
|pageCount|int|Общее число страниц.|
pageSize|int|Количество событий на запрошеной странице.|
